### RecurrentPPO Integration
- Added `sb3-contrib>=2.4.0` to requirements.txt
- Created `_build_full_obs_recurrent_ppo()` builder: uses `MlpLstmPolicy` with configurable `lstm_hidden_size` and `n_lstm_layers`, same PPO hyperparams as existing builder
- Created `RecurrentModelAdapter` class: wraps RecurrentPPO model to manage LSTM hidden states when used as opponent in `SingleAgentPEWrapper`. Tracks `lstm_states` and `episode_starts` across steps, provides `reset()` for episode boundaries
- Updated `AMSDRLSelfPlay.__init__()`: added `lstm_hidden_size` and `n_lstm_layers` params, updated `_model_class` dispatch for `"recurrent_ppo"`

- Updated `_get_build_kwargs()`: passes `lstm_hidden_size`, `n_lstm_layers`, `n_steps`, `batch_size` for recurrent_ppo
- Updated `_algo_class()`: returns `RecurrentPPO` for `"recurrent_ppo"`
- Updated `_wrap_opponent_model()`: wraps with `RecurrentModelAdapter` when algorithm is recurrent_ppo (before FixedSpeedModelAdapter if applicable)
- Updated `_evaluate_head_to_head_full_obs()`: added `recurrent` parameter for LSTM state tracking during evaluation — resets states per episode, passes states across steps
- Updated `_evaluate()`: passes `recurrent=(self.algorithm == "recurrent_ppo")` to eval function

## Decisions Made
| Decision | Rationale |
|----------|-----------|
| RecurrentPPO only in `--full_obs` mode | LSTM replaces the BiMDN/PartialObsWrapper approach to temporal memory. Combining both would be redundant and complex. |
| `RecurrentModelAdapter` wraps before `FixedSpeedModelAdapter` | LSTM state management must happen at the model.predict() level, before action expansion |
| `net_arch=dict(pi=[256,256], vf=[256,256])` for RecurrentPPO | Matches existing PPO [256,256] architecture, RecurrentPPO uses dict format for separate pi/vf networks |
